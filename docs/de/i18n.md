# Internationalisierung (i18n)

Xingluo wird mit zweisprachiger UI-Unterstützung (zh-CN / en) ausgeliefert und verwendet die Routing-Strategie `prefixDefaultLocale: false`, sodass die Standardsprache kein URL-Präfix hat.

## Routing-Strategie

Astros `i18n`-Konfiguration (siehe `astro.config.ts`):

```ts
i18n: {
  locales: ["zh-cn", "en"],
  defaultLocale: "zh-cn",
  routing: { prefixDefaultLocale: false },
}
```

**Wichtig: `prefixDefaultLocale: false` generiert nicht automatisch lokalisierte Seitenkopien** — Sie müssen `[locale]/`-Spiegelrouten manuell verwalten.

Xingluos Ansatz:

- **Root-Seiten** = Standardsprache (`zh-cn`), kein URL-Präfix, z. B. `/posts/welcome/`
- **`src/pages/[locale]/`** spiegelt alle Seiten; `getStaticPaths` verwendet `getLocaleParams()`, um nur nicht standardmäßige Locales zu generieren, z. B. `/en/posts/welcome/`
- Spiegelseiten sind ebenfalls dünne Wrapper, die dieselbe View-Komponente für die Renderlogik wiederverwenden

```
/                      → home (zh-cn)
/en/                   → home (en)
/posts/welcome/        → post (zh-cn)
/en/posts/welcome/     → post (en)
```

## Locale-Auflösung

View-Komponenten verwenden `Astro.currentLocale` zur automatischen Auflösung:

- Root-Seiten → `zh-cn`
- `[locale]`-Segment-Seiten → `en` (oder andere nicht standardmäßige Locales)

Es sind keine Pfadprüfungen auf der Komponentenebene erforderlich; `useTranslations(locale)` ruft die entsprechenden Zeichenfolgen direkt ab.

## i18n-Modulstruktur

[`src/i18n/`](../src/i18n/):

| Datei            | Zuständigkeit                                                                                                                              |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------ |
| `index.ts`       | `import.meta.glob("./lang/*.ts", {eager:true})` lädt Sprachen; exportiert `DEFAULT_LOCALE`, `LOCALES`, `useTranslations(locale)`, `tplStr` |
| `types.ts`       | Vollständige `UIStrings`-Schnittstelle (alle zu lokalisierenden Zeichenfolgen)                                                             |
| `routing.ts`     | `getLocalePrefix`, `withLocale(path, locale)`, `parseLocaleFromPath(pathname)`                                                             |
| `staticPaths.ts` | `NON_DEFAULT_LOCALES`, `getLocaleParams()`                                                                                                 |
| `format.ts`      | `tplStr(template, vars)` — `{{key}}` Platzhalter-Ersetzung                                                                                 |
| `lang/zh-cn.ts`  | Vereinfachtes Chinesisch (Standard)                                                                                                        |
| `lang/en.ts`     | Englisch                                                                                                                                   |

## UIStrings-Struktur

Die `UIStrings`-Schnittstelle definiert alle zu lokalisierenden UI-Zeichenfolgen, gruppiert organisiert:

- `nav`: Navigation (Startseite/Beiträge/Tags/Über/Archive/Suche/RSS)
- `post`: Beitrag (Datum, Teilen, Tags, Zurück, Bearbeiten, Inhaltsverzeichnis, Code kopieren, Bild-Lightbox, usw.)
- `pagination`: Seitennummerierung
- `home`: Startseite (Soziale Links, Hervorgehoben, Neueste)
- `archives`: Archiv (Anzahl, Monate)
- `footer`: Fußzeile (Urheberrecht)
- `pages`: Seitentitel und -beschreibungen
- `a11y`: Barrierefreiheitskennzeichnungen
- `languageSwitcher`: Sprachumschalter
- `notFound`: 404
- `comments`: Kommentarbereich

## Vorlagenzeichenfolgen

Zeichenfolgen mit Platzhaltern verwenden `{{key}}`, ersetzt über `tplStr`:

```ts
import { tplStr } from "@/i18n";

// archives.postCount = "{{count}} Beiträge"
tplStr(t.archives.postCount, { count: 5 }); // "5 Beiträge"
```

## Mehrsprachige SEO-Deklarationen

`Layout.astro`'s head gibt aus:

- `<link rel="alternate" hreflang="..." href="...">` für jede Sprache
- `x-default` zeigt auf die Standardsprache
- Die Sitemap-Integration aktiviert die i18n-Konfiguration, um hreflang automatisch zu generieren
- Beiträge mit nicht-standardmäßiger Sprache haben canonical, das auf das Original der Standardsprache verweist (um Duplicate-Content-Strafen zu vermeiden; siehe [SEO](./seo.md))

## Hinzufügen einer Sprache

Beispiel: Hinzufügen von Japanisch `ja`:

1. **`astro.config.ts`**: Fügen Sie `"ja"` zu `i18n.locales` und die Zuordnung `"ja": "ja-JP"` zu den Sitemap-`i18n.locales` hinzu
2. **`src/i18n/lang/`**: Erstellen Sie `ja.ts`, das ein vollständiges `UIStrings` exportiert (kopieren Sie `en.ts` und übersetzen Sie)
3. **`src/i18n/staticPaths.ts`**: `NON_DEFAULT_LOCALES` enthält automatisch `ja` (berechnet aus `LOCALES`)
4. **`src/pages/[locale]/`**: Spiegelseiten generieren automatisch die `ja`-Version (`getLocaleParams` deckt es ab)
5. **Sprachumschalter**: Fügen Sie `"ja": "日本語"` zu `languageSwitcher.names` in `zh-cn.ts` und `en.ts` hinzu

## Übersetzung auf Inhaltsebene

Xingluo unterstützt mehrsprachige Beitragsinhalte über die Frontmatter-Felder `locale` und `translationKey`.

### Grundlegende Verwendung

1. **Beitrag in Standardsprache**: Platzieren Sie ihn in `src/content/posts/<slug>.md`, setzen Sie `translationKey` als Gruppenkennung:

```yaml
# src/content/posts/welcome.md
---
title: "欢迎来到星罗"
locale: zh-cn
translationKey: welcome-to-xingluo
tags: [公告, Astro]
---
```

2. **Übersetzung**: Platzieren Sie sie in einem Sprachunterverzeichnis `src/content/posts/<locale>/<slug>.md` mit demselben `translationKey`:

```yaml
# src/content/posts/en/welcome.md
---
title: "Welcome to Xingluo"
locale: en
translationKey: welcome-to-xingluo
tags: [announcement, Astro]
---
```

### Verzeichnisstruktur

```
src/content/posts/
├── welcome.md              # Standardsprache (zh-cn)
├── en/
│   └── welcome.md          # Englische Übersetzung
├── ja/
│   └── welcome.md          # Japanische Übersetzung
└── another-post.md         # Unabhängiger Beitrag (kein translationKey)
```

- Die Namen der Sprachunterverzeichnisse müssen mit den Sprachcodes in `i18n.locales` von `astro.config.ts` übereinstimmen
- Sprachunterverzeichnisse werden aus dem URL-Slug herausgefiltert (z. B. `/posts/welcome/`, nicht `/posts/en/welcome/`)
- Beiträge ohne `translationKey` sind unabhängig und nicht sprachübergreifend verknüpft

### Routing-Verhalten

| Szenario                                          | Verhalten                                                                                        |
| ------------------------------------------------- | ------------------------------------------------------------------------------------------------ |
| Standard-Locale-Zugriff auf einen `zh-cn`-Beitrag | Rendert das Original in der Standardsprache                                                      |
| Nicht standardmäßige Locale mit **Übersetzung**   | Rendert die entsprechende Übersetzung                                                            |
| Nicht standardmäßige Locale **ohne** Übersetzung  | Fällt auf das Original in der Standardsprache zurück (identischer Inhalt, canonical schützt SEO) |

### Listendeduplizierung

Listenseiten (Startseite, Beitragsliste, Tags, Archive, RSS) verwenden `getPostsForLocale`, um repräsentative Beiträge pro Sprache auszuwählen: Jede Übersetzungsgruppe zeigt nur eine Karte in der Zielsprache an und verhindert doppelte Einträge für dasselbe Thema.

### Canonical und SEO

- **Hat eine unabhängige Übersetzung**: canonical zeigt auf die eigene URL der Übersetzung, von Suchmaschinen separat indexierbar
- **Keine Übersetzung (Fallback)**: canonical zeigt auf das Original in der Standardsprache, um Duplicate-Content-Strafen zu vermeiden
- hreflang-Deklarationen decken alle Sprachen ab und informieren Suchmaschinen über die Beziehungen zwischen den Sprachversionen

Siehe [SEO](./seo.md).
